## Overview

This hub is the site you are reading: a deliberately small platform for **projects** (productized systems), **work** (engagements), **writing** (longer judgment), and **labs** (experiments). The goal is not to maximize components—it is to keep every entry honest, typed, and easy to evolve.

## Problem

Most personal sites mix blog posts, “projects,” and client work in one flat stream. That reads fine for a few items and falls apart when:

- the same engagement is described differently on the home page and on a detail page;
- frontmatter drifts from what the UI actually shows;
- you add a fourth kind of content and copy-paste a fifth template.

I wanted a structure where **discipline is default**: invalid content fails the build, and each domain has a clear job.

## Solution

The first implementation stayed boring on purpose: **Next.js App Router**, **MDX** with **gray-matter** for frontmatter, and TypeScript types that mirror what `parse.ts` enforces. Later iterations kept the typed contracts but shifted writing and projects to DB/admin-managed flows.

Nothing here tries to be a headless CMS, a design system, or a novel renderer. It is a **content-first spine** you can ship and then improve in place.

## Architecture

At a high level:

- **V1 content** lived under `src/content/{projects,work,writing,labs}/` as `.mdx` files.
- **Parsing** turns each file into a typed entry plus a raw body string for MDX.
- **Routes** map slugs to those entries; metadata and JSON-LD helpers read the same shapes.

That split means you can change typography or layout without touching the ingest path, and you can tighten schemas without redrawing every page.

## Key Decisions

- **Strict over flexible:** Optional fields exist, but there is no “grab bag” frontmatter. If it is not in the type, it does not belong.
- **Published vs draft:** Only `status: published` or omitted status ships publicly; drafts stay out of feeds and static generation lists.
- **Featured is scarce:** Homepage blocks pull `featured: true` entries. The default is off so “featured” still means something in a year.

## Outcomes

What I optimize for with this stack:

- **Clarity:** Readers see consistent sections and metadata whether they land on a list or a detail page.
- **Sustainability:** New content is “fill in the template + review,” not “invent a layout.”
- **Honest scope:** Labs can stay rough; work can stay anonymized; projects can stay product-shaped—without blurring those lines.

The next improvements are incremental: richer intros that surface more structured fields, or JSON-LD that reuses `problem` / `solution` where it helps search—not a rewrite of the model.
